🔌 Step-by-Step API Integration Guides

How to integrate each POS device provider with your billing software — verified from official documentation

How POS Device Integration Works Your billing software (elPOS) runs on a tablet/laptop. When the customer pays, elPOS sends a payment request to the EDC/POS device via API. The device handles the card tap/swipe or UPI scan, then sends the result back. Your billing software never touches card data — the device handles all the sensitive stuff.

1. Pine Labs — Cloud Integration

REST API • No local cable needed • Works over internet

Source: developer.pinelabs.com/in/instore/cloud-integration

Prerequisites

  • Pine Labs merchant account with MerchantID and SecurityToken (UUID format)
  • Pine Labs EDC terminal deployed at your store
  • Internet connectivity on both billing device and terminal

Base URLs

UAT (Testing): https://www.plutuscloudserviceuat.in:8201/
Production: Provided by Pine Labs on go-live

Step-by-Step Flow

1
Send Payment Request

Your billing app calls UploadBilledTransaction API with amount, payment mode, and transaction number. Pine Labs returns a PTRID (Plutus Transaction Reference ID).

POST /UploadBilledTransaction
{
  "TransactionNumber": "INV-2025-001",
  "SequenceNumber": 1,
  "AllowedPaymentMode": "1",     // 1=Card, 10=UPI
  "Amount": 150000,               // In paisa (₹1500.00)
  "MerchantID": 1234,
  "SecurityToken": "70D7509C-0A90-4938-...",
  "StoreId": 60588,
  "AutoCancelDurationInMinutes": 5
}

// Response
{
  "ResponseCode": 0,
  "ResponseMessage": "APPROVED",
  "PlutusTransactionReferenceID": 501
}
2
Customer Pays on Terminal

Cashier enters the PTRID on the Pine Labs terminal (or the terminal automatically picks it up). Customer taps/inserts card or scans UPI. The terminal processes the payment with the bank.

3
Check Payment Status

Your billing app polls GetStatus API with the PTRID. Alternatively, Pine Labs can push the result to your PostBack URL (webhook).

POST /GetStatus
{
  "MerchantID": "1234",
  "SecurityToken": "70D7509C-...",
  "StoreID": "12301001",
  "PlutusTransactionReferenceID": 501
}

// Response
{
  "ResponseCode": 0,
  "ResponseMessage": "TXN APPROVED",
  "TransactionData": [
    {"Tag": "PaymentMode", "Value": "CARD"},
    {"Tag": "Amount", "Value": "1500"},
    {"Tag": "RRN", "Value": "000792514130"},
    {"Tag": "ApprovalCode", "Value": "849035"},
    {"Tag": "CardNumber", "Value": "XXXXXXXXXXXX1234"}
  ]
}
4
Mark Sale Complete

On success, mark the sale as paid in your billing system. Print receipt. To cancel a pending payment, use CancelTransaction API.

Payment Mode Codes

1 = Card 10 = UPI 6 = Brand EMI 20 = Bank EMI 11 = Bharat QR 21 = Amazon Pay

Other APIs

CancelTransaction ForceCancel Void/Refund (TxnType=1) Split Payment (SequenceNumber) GST fields supported

2. Razorpay POS — Android SDK (Ezetap)

Android SDK • On-device or Bluetooth • Universal Pay API

Source: Razorpay POS Payment SDK (GitHub)

Prerequisites

  • Razorpay POS merchant account — contact Razorpay team for API keys
  • Demo and Production app keys
  • Android billing app (SDK is Android-native, Java/Kotlin)
  • Razorpay POS device (Android Smart POS, Mini POS, or external card reader via Bluetooth/USB)

Step-by-Step Flow

1
Add SDK Dependency 
// build.gradle (project)
maven { url 'https://jitpack.io' }

// build.gradle (app)
implementation 'com.github.AtifQEzetap:razorpay-pos-payment-sdk:2.2'
2
Initialize SDK

Call EzeAPI.initialize() once on app startup. This connects to Razorpay servers and sets up the device.

InitializeRequest req = new InitializeRequest();
req.setDemoAppKey("your_demo_key");    // or setProdAppKey()
req.setMerchantName("Your Store");
req.setUserName("merchant_id");
req.setCurrencyCode("INR");
req.setCommunicationChannel("BT");     // BT, USB, or NONE

EzeAPI.initialize(activity, REQUEST_CODE, req);
3
Prepare Device

Initialize the card reader with latest encryption keys from the bank.

EzeAPI.prepareDevice(activity, REQUEST_CODE);
4
Process Payment (Universal Pay API)

Single API call for all payment modes — card, UPI, wallet. The SDK handles the device interaction.

GenericPayment payment = new GenericPayment();
payment.setAmount(150000);              // In paisa (₹1500.00)
payment.setTxnType("PAY");
payment.setPaymentMode("CARD");         // or "UPI"

// Optional: add GST, customer info, references
payment.setGst(sgst, cgst, igst, cess);
payment.setCustomer(name, email, mobile);
payment.setReference1("INV-2025-001");

EzeAPI.performTransaction(activity, REQUEST_CODE, payment);
5
Handle Response

Result arrives in onActivityResult(). Contains transaction ID, RRN, card last 4, approval code.

Other APIs

EzeAPI.getCardInfo() EzeAPI.sendReceipt() EzeAPI.serviceRequest() GST fields supported 7 reference fields for context

Step-by-Step Flow

1
Send Sale Request

Billing system sends payment request to Paytm POS device with transaction amount, currency, and unique reference number.

2
Customer Pays on Device

Device shows card insert/tap prompt with available payment options. Customer enters PIN and confirms.

3
Middleware Processes & Returns Result

Paytm middleware processes the transaction. Success/failure response displays on device and returns to your billing system.

4
Verify with Status API

Call Status Enquiry API to confirm the transaction. Use Void API to cancel if needed.

Available APIs

4. PhonePe — Collect Call (Request Payment API)

REST API • Server-to-server • Push payment to customer's phone

Source: developer.phonepe.com — Request Payment API

Prerequisites

  • PhonePe merchant account with merchantId, saltKey, and saltIndex
  • Customer's mobile number (payment request is sent to their PhonePe app)

Base URLs

UAT: https://mercury-uat.phonepe.com/enterprise-sandbox/v3/charge
Production: https://mercury-t2.phonepe.com/v3/charge

Authentication

X-VERIFY: SHA256(base64_payload + '/v3/charge' + saltKey) + '###' + saltIndex

Step-by-Step Flow

1
Send Payment Request 
POST /v3/charge
Headers:
  Content-Type: application/json
  X-VERIFY: {checksum}###1
  X-CALLBACK-URL: https://yoursite.com/webhook  (optional)

// Payload (base64 encoded in "request" field)
{
  "merchantId": "YOUR_MERCHANT_ID",
  "transactionId": "TXN-2025-001",      // Unique, <35 chars
  "merchantOrderId": "INV-2025-001",
  "amount": 150000,                       // In paisa (₹1500)
  "instrumentType": "MOBILE",
  "instrumentReference": "9876543210",    // Customer phone
  "storeId": "STORE-01",
  "expiresIn": 300                        // 5 min timeout
}

// Response
{
  "success": true,
  "code": "SUCCESS",
  "data": {
    "transactionId": "TXN-2025-001",
    "amount": 150000,
    "merchantId": "YOUR_MERCHANT_ID"
  }
}
2
Customer Gets Notification

PhonePe sends a payment request to the customer's phone. Customer opens PhonePe app, sees the amount, and confirms payment.

3
Receive Callback or Poll Status

PhonePe sends webhook to your X-CALLBACK-URL on success/failure. For pending transactions, poll the status API:

GET /v3/transaction/{merchantId}/{transactionId}/status

Key Response Codes

SUCCESS — Request sent AUTHORIZATION_FAILED — Bad checksum BAD_REQUEST — Missing params INVALID_TRANSACTION_ID — Duplicate

6. Cashfree — SoftPOS (Tap to Pay)

Android App/SDK • NFC Tap & Pay • No hardware needed

Source: cashfree.com/soft-pos

How SoftPOS Works

No separate hardware device needed. Cashfree SoftPOS is an Android app/SDK that turns any NFC-enabled Android phone into a card payment terminal. Customer taps their card on the merchant's phone.

Device Requirements

  • Tap & Pay (NFC): Android 9 (Pie) or above — phones from 2018+
  • Other methods (QR, links): Android 6 (Marshmallow) or above
  • NFC hardware required on the phone for card tap

Supported Payment Methods

Card Tap & Pay (NFC) UPI QR Code Payment Links Wallets EMI / BNPL
Integration Note Cashfree SoftPOS is distributed as a standalone app or embeddable SDK. For API-level integration with billing software, refer to the Cashfree Payments API documentation. The SoftPOS SDK handles the NFC card read locally and processes via Cashfree's payment gateway.

7. BharatPe — No Public API

Closed ecosystem • UPI QR focus • Merchant Plugin model

No Third-Party API Available BharatPe does not offer a publicly available API for third-party billing software integration. Their POS devices (BharatPe One, SwipeX) operate as closed systems. Payment processing is handled entirely within the BharatPe app ecosystem.

What BharatPe Offers Instead

  • Merchant Plugin: In collaboration with Unity Bank, BharatPe offers a UPI Plug-In that merchants can embed in their own apps to accept UPI payments without redirecting to a third-party app
  • QR Code Payments: Static and dynamic QR — works with all UPI apps (GPay, PhonePe, Paytm, etc.)
  • BharatPe One Device: Standalone all-in-one POS with built-in billing — not designed to integrate with external billing software

For elPOS: BharatPe is not suitable for API integration. Use Pine Labs, Razorpay, or Paytm for card payments, and PhonePe or standard UPI QR for UPI payments instead.

🏦 Bank-Provided EDC Terminals

Banks that deploy EDC machines to their merchant account holders — these are bank-locked devices sourced from OEMs like Ingenico, Verifone, and PAX

How Bank EDC Differs from Aggregator POS Banks provide EDC terminals as part of their merchant acquiring service. The device is tied to the merchant's bank account — payments settle directly into it. Unlike aggregators (Pine Labs, Razorpay), bank EDC devices generally do not offer open APIs for third-party POS software integration. The bank controls the software on the device.

Key Difference: Bank EDC vs Aggregator POS Bank terminals are closed systems — they only process payments and settle to that bank's account. They cannot be integrated with third-party billing software like elPOS via API. For elPOS integration, use aggregator providers (Pine Labs, Razorpay/Ezetap, etc.) which offer open APIs and push-to-device models that work alongside any billing system.
