Requirement Types

Every incoming request falls into one of three categories

New Feature

Entirely new functionality that does not currently exist in the system. Requires the most thorough discovery and definition process.

Feature Enhancement

An improvement or modification to an existing feature. Builds on established behavior with targeted changes.

Bug

A defect in existing behavior where the system does not work as defined. Focused on reproducing, diagnosing, and correcting.

The depth of the "Know Your Requirement" phase varies by type — new features require the most discovery, bugs the least.

Phase 1: Know Your Requirement

Understand the problem before jumping to solutions

Who Raises It

Requirements can originate from business stakeholders (client-side) or internal team members. Every requirement must have a named owner responsible for providing context.

Mandatory Intake Sheet

Before any work begins, the team must have clear answers to these five questions. It is our responsibility to extract this information from the client — through conversations, calls, or emails — and document it. The requirement is not ready until all five fields are answered.

No answers to these five → no work starts.

Once confirmed by the client, a summary is sent via Email + WhatsApp as the first client checkpoint — "Here's what we understood."

Phase 2: Definitions

Define what the system will — and will not — do

System Behavior Definitions

A clear written statement of what the implementation will do and, equally important, what it will not do. This eliminates ambiguity and prevents scope drift.

Scope Boundaries

Explicit declaration of what is included and excluded. If it is not listed in scope, it is not part of this implementation.

Acceptance Criteria

Measurable conditions that must be met for the requirement to be considered complete. Written in a testable format.

Time Estimate

Estimated hours for implementation, reviewed and agreed upon before work begins. This becomes the baseline for tracking.

Phase 3: Implementation

Build on the Alpha environment with tracked effort

Alpha Environment

All development work happens in the Alpha environment — an internal, non-production space used for active development and internal testing.

Estimated Hours

Hours from the Definitions phase are used as the working budget. Any deviation is flagged immediately.

Phase 4: Change Request

A controlled process for mid-implementation changes

When Does a Change Request Occur?

During implementation, the team may discover a better approach, gain new insight, or identify a necessary scope adjustment. Rather than absorbing changes silently, every change goes through this process.

Trigger

  • New insight discovered during development
  • A better technical or UX approach identified
  • Scope adjustment required due to unforeseen dependencies

Impact Analysis

DimensionAssessment
Time impactAdditional hours required
Scope impactWhat changes in the deliverable
Risk assessmentPotential side effects or regressions

Communication

Stakeholders are notified via Email WhatsApp with the change details and impact summary.

Approval

No change is absorbed without explicit client/stakeholder sign-off. Work pauses on the changed item until approval is received.

Timeline Adjustment

Upon approval, hours and delivery date are updated. The change feeds back into the Implementation phase with revised parameters.

Phase 5: Beta & Usability Testing

Validate with real data and real users before release

Beta Environment

The Beta environment mirrors production data. This ensures testing happens under realistic conditions, not idealized dev data.

Usability Test Definition

  • What we test — Specific workflows and scenarios derived from acceptance criteria
  • Who tests — Identified test users (internal QA + selected end users)
  • Success metrics — Task completion, error rate, user feedback

Usability Report

A summary of findings from testing: what worked, what didn't, and what was changed as a result.

Changes Log

Every change made during Beta is logged with reason and timestamp. Nothing is changed without documentation.

Phase 6: Training

Ensure users know how to use what was built

Training Sessions

Conducted on the Beta environment so users interact with near-production functionality. Training covers new workflows, changed behaviors, and edge cases.

Target Audience

Identified user groups who will directly use the new or changed functionality. Training is tailored to each group's role.

Phase 7: Release

Deploy to production with clear communication

Release to Production

After training is complete and sign-off is received, the implementation is deployed to the production environment.

Release Notes

A summary of what was released — new features, enhancements, or fixes — shared with all stakeholders via Email WhatsApp.

Release Date

The confirmed date of production deployment, communicated in advance to allow stakeholders to prepare.

Phase 8: Post-Release Monitoring

Watch for issues before closing the loop

Monitoring Phase

A defined observation period after release to catch issues that only surface with real production usage and load.

What We Monitor

  • System errors and exceptions related to the release
  • User-reported issues
  • Performance metrics in affected areas
  • Adoption and usage patterns

Communication Protocol

Consistent, predictable communication at every phase

Channels

All client communication is shared via Email and WhatsApp to ensure visibility across stakeholders.

3 Client Checkpoints

Client communication is structured into three clear checkpoints. Each checkpoint requires client acknowledgement before the process moves forward.

1

"Here's what we understood"

After Phase 1 — Know Your Requirement

Summary of the problem, its impact, and the desired outcome. The client confirms that the understanding is correct. Once confirmed, a formal acknowledgement is sent via Email + WhatsApp before any solution work begins.

2

"Here's what we will build"

After Phase 2 — Definitions

Scope, behavior definitions, acceptance criteria, and time estimate. The client approves the plan before build starts.

3

"Here's what changed after release"

After Phase 8 — Monitoring

Release outcomes, any post-release changes, and the closure summary. The client receives a complete picture of what was delivered.

Communication channels remain Email + WhatsApp for all checkpoints. Internal phase transitions are tracked but do not require separate client notifications.
