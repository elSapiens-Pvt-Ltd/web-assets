08 — Ledger Design

Immutable append-only ledger, balance computation, reconciliation, and expiry processing

Principle

Immutable Ledger Principle

Every financial event in elLoyalty is recorded as an immutable ledger entry. Balances are never stored directly — they are computed from the ledger. This provides a complete audit trail and makes reconciliation trivial.

No Updates, No Deletes

The ledger_entries and wallet_transactions tables have no UPDATE or DELETE operations. Corrections are made by appending new entries of type adjust or reverse.

Balance = SUM(entries)

A customer's coin balance = SUM of all coin entries. A materialized view pre-computes this for read performance. Refreshed on every write.

Complete Audit Trail

Every entry links back to its cause: sale_id, refund_id, topup_id, or admin adjustment. Full traceability from any balance to its origin.

Balance

Balance Computation

Materialized View

-- Materialized view for fast balance reads CREATE MATERIALIZED VIEW wallet_balances AS SELECT w.id AS wallet_id, w.customer_id, w.organization_id, w.workspace_id, COALESCE(SUM(wt.amount) FILTER ( WHERE wt.sub_account = 'coins' ), 0) AS coin_balance, COALESCE(SUM(wt.amount) FILTER ( WHERE wt.sub_account = 'cashback' AND wt.type = 'earn' AND wt.created_at > now() - interval '24 hours' -- pending window ), 0) AS cashback_pending, COALESCE(SUM(wt.amount) FILTER ( WHERE wt.sub_account = 'cashback' AND (wt.type != 'earn' OR wt.created_at <= now() - interval '24 hours') ), 0) AS cashback_ready, COALESCE(SUM(wt.amount) FILTER ( WHERE wt.sub_account = 'cash' ), 0) AS cash_balance FROM wallets w LEFT JOIN wallet_transactions wt ON wt.wallet_id = w.id GROUP BY w.id, w.customer_id, w.organization_id, w.workspace_id; CREATE UNIQUE INDEX idx_wb_wallet ON wallet_balances (wallet_id); -- Refresh on every transaction (or via trigger) REFRESH MATERIALIZED VIEW CONCURRENTLY wallet_balances;

Dual-Write: Cached + Computed

For read performance, elLoyalty uses a dual strategy:

  1. Hot path (Redis): Balance cached in Redis with key wallet:{id}:balance. Updated atomically on every transaction using Redis INCRBY/DECRBY.
  2. Cold path (PostgreSQL): Materialized view refreshed periodically (every 5 minutes) or on-demand via REFRESH MATERIALIZED VIEW CONCURRENTLY.
  3. Reconciliation: Background worker compares Redis balances with materialized view every 15 minutes. Discrepancies trigger alerts and auto-correction from the ledger (source of truth).

  wallet_id: w_001, type: earn, sub_account: cashback
  amount: +4000 (₹40), reference_type: sale, reference_id: sale_001
  balance_after: ₹40 pending

═══════ T+1: Cashback settles ═══════
  (No new entry — the pending window expires and the materialized view
   reclassifies it from pending to ready on next refresh)

═══════ REFUND: Full refund of sale_001 ═══════

Entry #5: REVERSE coin redemption
  type: reverse, sub_account: coins
  amount: +200, reference_type: refund, reference_id: refund_001

  type: reverse, sub_account: cash
  amount: +10000, reference_type: refund, reference_id: refund_001

  type: reverse, sub_account: coins
  amount: -80, reference_type: refund, reference_id: refund_001

Entry #8: REVERSE cashback
  type: reverse, sub_account: cashback
  amount: -4000, reference_type: refund, reference_id: refund_001
  balance_after: ₹0  ← back to original!
Expiry

Expiry Processing

Coins and cashback can have configurable expiry periods. A background worker processes expirations:

// services/expiry_svc.go — runs every hour via cron func (s *ExpiryService) ProcessExpiredEntries(ctx context.Context) error { // Find all unprocessed expiry events where expires_at <= now() events, err := s.repo.FindPendingExpiries(ctx, time.Now()) if err != nil { return err } for _, evt := range events { tx, _ := s.pool.Begin(ctx) // Check current balance — don't expire more than available balance := s.walletSvc.GetSubAccountBalance(ctx, evt.WalletID, evt.SubAccount) expireAmt := min(evt.Amount, balance) if expireAmt > 0 { // Create debit entry s.ledgerSvc.Record(ctx, tx, LedgerEntry{ WalletID: evt.WalletID, CustomerID: evt.CustomerID, Type: TxExpire, SubAccount: evt.SubAccount, Amount: -expireAmt, RefType: "expiry", RefID: evt.ID, Description: fmt.Sprintf("Expired %d %s", expireAmt, evt.SubAccount), }) } // Mark event as processed s.repo.MarkProcessed(ctx, tx, evt.ID) tx.Commit(ctx) // Notify customer s.bus.Publish(ctx, "coins.expired", CoinsExpiredEvent{...}) } return nil }

Reconciliation

Reconciliation with elbooks

Wallet top-ups and redemptions that affect real money must be synced to elbooks for accounting:

  • Top-up: When customer adds ₹500 to wallet → elLoyalty creates an earn entry → publishes wallet.topup event → elbooks creates a "Loyalty Liability" journal entry
  • Redemption: Customer pays ₹200 from wallet → elLoyalty creates a redeem entry → elbooks reduces the liability and records it as a discount on the invoice
  • Expiry: ₹50 in cashback expires → elLoyalty creates an expire entry → elbooks reverses the liability (income recognition)
  • Reconciliation report: Daily automated report comparing elLoyalty ledger totals with elbooks GL balances. Discrepancies flagged for manual review.
