03 — Checkout Flow

End-to-end integration of elLoyalty with elPOS checkout — from cart evaluation to post-sale earning

Overview

Checkout Sequence Overview

The checkout flow has three phases: Evaluate (before payment), Apply (with payment), and Earn (after payment). Frontends (POS UI, mobile) use the GraphQL BFF for phases 1–2. The earn phase is typically backend-to-backend via gRPC.

Phase 1

Evaluate Phase

Triggered when a customer is attached to the cart, or when the cart changes after customer attachment.

1

POS attaches customer

Operator scans loyalty card, enters phone number, or selects from recent customers. POS now has customer_id.

2

POS calls benefit evaluation

Via GraphQL BFF (preferred): query { benefitPreview(customerId, cartItems) { ... } } — a single query can also fetch the customer wallet and coupons in the same round-trip. Via REST: POST /api/v1/benefits/evaluate. Both accept cart items (product IDs, quantities, prices, categories), customer ID, and store context.

3

Engine gathers context

Loads customer profile (tier, tags), wallet balances, active rules, active campaigns, and outstanding coupons — all with Redis caching.

4

Providers evaluate independently

Each provider (Coins, Cashback, Wallet, Coupon, Campaign) evaluates the cart against its rules. Returns []CandidateBenefit.

5

Conflict resolution + scoring

Engine scores candidates, resolves conflicts (exclusive vs stackable), and returns EvaluationResult with recommended selections.

6

POS displays benefit panel

Customer sees available benefits: "Use 200 coins (₹20 off)", "10% cashback", "Apply coupon SAVE50". Recommended benefits are pre-checked.

Re-evaluation Triggers

The POS must re-call the evaluation (GraphQL benefitPreview query or REST /evaluate) whenever the cart changes after initial evaluation:

  • Item added or removed
  • Quantity changed
  • Payment method changed (some benefits are payment-method-specific)
  • Coupon code entered

The engine debounces evaluations — a 500ms cooldown on the POS side prevents rapid re-calls during barcode scanning.

Phase 2

Apply Phase

Triggered when the customer confirms their benefit selections and the operator proceeds to payment.

1

Customer confirms selections

Customer toggles benefits on/off. POS collects the list of selected_benefit_ids.

2

POS calls benefit apply

Via GraphQL BFF (preferred): mutation { redeemBenefits(input: { selectedBenefitIds, cartItems, ... }) { ... } }. Via REST: POST /api/v1/benefits/apply. Both send the selected_benefit_ids. This is a write operation.

3

Engine re-validates

Re-runs evaluation to ensure benefits are still valid (anti-race condition). A coupon may have been used by another terminal in the interval.

4

Atomic transaction

Within a single PostgreSQL transaction: debit wallet (if coins/cash used), mark coupon as redeemed, update campaign counters, write ledger entries.

5

Return discount breakdown

Returns total discount amount, per-benefit breakdown, and adjusted cart total. POS uses this for payment screen.

Apply Response

{ "data": { "application_id": "app_uuid", "total_discount": 5000, // ₹50.00 in paisa "applied": [ { "benefit_id": "ben_coins_200", "type": "loyalty_coins", "discount": 2000, // ₹20 from 200 coins "coins_debited": 200 }, { "benefit_id": "ben_coupon_save50", "type": "coupon", "discount": 3000, // ₹30 from coupon SAVE50 "coupon_code": "SAVE50" } ], "adjusted_total": 95000 // ₹950/₹1000 after ₹50 discount } }
Phase 3

Earn Phase

Triggered after successful payment. Records what the customer earned from this sale.

1

elPOS confirms payment

After the sale is finalized, elPOS backend calls gRPC: CreditEarnings (or REST POST /api/v1/benefits/earn) with the completed sale data. This is a backend-to-backend call — it does not go through the GraphQL BFF.

2

Calculate earning

Engine evaluates earning rules: X coins per ₹Y spent, cashback percentage, bonus campaign rewards. Earning is calculated on the post-discount total.

3

Credit coins immediately

Loyalty coins are credited to the customer's wallet instantly. They can be used in the next transaction.

4

Queue delayed cashback

Cashback is queued with the configured delay (instant, T+1, T+7). A background worker processes the queue and credits wallets when the delay expires.

5

Print earn summary on receipt

elPOS prints on the receipt: "You earned 95 coins! Total balance: 345 coins. ₹9.50 cashback pending."

Earn Request

{ "sale_id": "sale_uuid", "customer_id": "cust_uuid", "sale_total": 95000, // ₹950 post-discount, paisa "payment_method": "upi", "items": [...], // same CartItem array "application_id": "app_uuid", // link to benefits applied "store_id": "store_uuid", "terminal_id": "term_uuid", "operator_id": "person_uuid" }

Earn Response

{ "data": { "coins_earned": 95, "coin_balance": 345, "cashback_amount": 950, // ₹9.50 pending "cashback_available_at": "2026-04-03T00:00:00Z", "bonus_coins": 0, "message": "You earned 95 coins! ₹9.50 cashback will be available tomorrow." } }
Offline

Key Differences in Offline Mode

Pre-Allocated Budget

During last sync, elLoyalty sends a "coin budget" and "wallet budget" — max amounts the terminal can debit without server confirmation. Prevents double-spend.

Cached Rules Only

Only rules cached during last sync are available. New campaigns or coupons created while offline won't appear until sync.

Approximate Earning

Coins earned offline are estimated and displayed on receipt with "~" prefix. Exact earning is calculated on reconciliation.

No Coupon Validation

Coupon codes entered offline cannot be validated against usage limits. They are queued and validated on sync — may be rejected post-facto.

Refund

Refund & Reversal Flow

When a sale is refunded, the applied benefits must be reversed:

GST

GST Compliance for Benefits

India's GST rules require that discounts be applied before tax calculation (pre-tax discount) or explicitly listed as post-tax adjustments. elLoyalty follows the pre-tax approach:

  • Benefits reduce the taxable value — GST is calculated on the discounted amount
  • This matches CBIC guidelines: discounts known at time of supply are deducted from transaction value
  • The apply response includes discount breakdown per item for SST/GST invoice line-item adjustments
  • Cashback earned post-sale is NOT a discount — it's a separate loyalty credit, no GST impact on original sale
  • Free items from campaigns (BOGO) are treated as samples with the applicable GST on deemed supply
