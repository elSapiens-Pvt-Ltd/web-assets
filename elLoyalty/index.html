Rationale

Why a Unified Benefits Platform?

Separate systems for coupons, points, cashback, and wallets create conflicting discounts, double-spend vulnerabilities, inconsistent balances, and engineering burden. A unified engine eliminates all of these.

The Problem with Separate Systems

When coupons live in marketing, points live in CRM, and cashback lives in finance — no single system can answer: "What is the best deal for this customer on this cart?" Conflicts go undetected. Stacking rules are unenforceable. Fraud vectors multiply.

Unified Engine Advantage

One service evaluates all benefit types atomically. It knows every rule, every wallet balance, every coupon constraint, and every campaign condition. It resolves conflicts, prevents double-spend, and returns the optimal combination — in a single API call.

Ecosystem Citizen

elLoyalty follows the elsapiens pattern: gRPC for inter-service calls, JWT from elAuth, financial sync to elbooks, shared multi-tenancy model (Reseller → Workspace → Organization → Customer). It is a first-class microservice, not a bolt-on.

Benefit Types

Five Distinct Reward Mechanisms

Each type has different earning rules, redemption constraints, expiry behavior, and financial impact. The Benefit Engine evaluates all of them together.

🪙 Loyalty Coins

Points earned on purchases. No direct monetary value until redeemed.

  • Configurable earn rate per org
  • Point-to-currency conversion
  • Expiry after N days of inactivity
  • Tier multipliers (Silver 1.5×, Gold 2×)
💵 Cashback

Post-purchase reward credited after a holding period. Usable on future purchases.

  • Percentage or flat amount
  • Holding period (24h–30d)
  • Category-specific rates
  • Cap per transaction
👛 Loyalty Cash

Pre-loaded wallet money. Direct monetary value. Acts as a payment method.

  • Top-up by customer or org
  • 1:1 currency value
  • Expiry configurable
  • Refund to wallet option
🎟️ Coupons

Rule-based discounts — percentage or flat. Single-use or multi-use, targeted or public.

  • Min purchase requirement
  • Product/category restrictions
  • User segment targeting
  • Usage limits (per user, global)
🎯 Campaigns

Event-driven rewards triggered by actions: signup, birthday, referral, spend milestones.

  • Time-bound activation
  • Trigger conditions
  • Reward any benefit type
  • Budget caps

Ecosystem

elsapiens Integration Map

elLoyalty connects to 5 ecosystem services via gRPC, following the same patterns as elPOS. A GraphQL BFF gateway aggregates these services for frontend consumers.

elAuth
Identity & Multi-Tenancy

JWT validation, user/org identity, workspace membership. elLoyalty validates JWTs using the shared HS256 secret. Tenant context (reseller_id, workspace_id, organization_id) extracted from claims.

elPOS
Primary Consumer

POS calls elLoyalty during checkout to evaluate benefits, then calls again to finalize redemption. Offline POS uses pre-cached budgets. Sale events trigger earn flows.

elbooks
Financial Sync

Loyalty redemptions are recorded as deductions in elbooks invoices. Cashback credits are journal entries. Monthly reconciliation between loyalty ledger and accounting ledger.

elCatalog
Product Context

Category and product data for rule evaluation. "Buy product X, get 2× coins" needs product lookup. Category-based coupon restrictions validated against catalog.

elmessagehub
Notifications

WhatsApp/SMS for: points earned, cashback credited, coupon expiry reminders, tier upgrades, campaign alerts. Triggered by domain events.

Isolation

Multi-Tenancy Model

Identical to the ecosystem standard — 4-tier hierarchy with row-level isolation via composite keys.

Reseller (white-label partner)
  └── Workspace (isolated business unit)
       └── Organization (entity/company)
            └── Customer (individual)
                 ├── Wallet (coins, cashback, loyalty cash)
                 ├── Coupon Assignments
                 ├── Campaign Enrollments
                 └── Transaction Ledger

Every query is scoped:
  WHERE organization_id = $1 AND workspace_id = $2

JWT Claims:
  {
    "entityId":     "org-uuid",       → organization_id
    "resellerId":   "reseller-uuid",  → reseller_id
    "workspaceIds": ["ws-uuid-1"],    → workspace_id (selected)
    "personId":     "user-uuid"       → operator identity
  }
Technology

Technology Stack

Backend

Go 1.24+ with Clean Architecture (Handler → Service → Repository). Chi v5 router, pgx v5 PostgreSQL driver, go-redis v9, gRPC with circuit breakers (sony/gobreaker), shopspring/decimal for money, zap structured logging.

Database

PostgreSQL 16+ primary store. UUID PKs, BIGINT for money (paisa), TIMESTAMPTZ, soft deletes, JSONB for rules. Redis 7+ for caching (wallet balances, rule evaluation cache, rate limits) and pub/sub events.

Communication

gRPC for ecosystem inter-service calls (elAuth, elPOS, elbooks, elCatalog, elmessagehub). GraphQL BFF (Go + gqlgen, port 4000) for frontend consumers — POS UI, admin dashboard, mobile app. REST for webhooks, sync, and health checks. Redis pub/sub for async events (earn, redeem, expire).

Frontend

React 18+ with elsapiens SDK, Zustand state management, Tailwind CSS + el-* classes. Admin dashboard for rules/campaigns. POS integration widget for checkout flow. Customer-facing wallet view.
