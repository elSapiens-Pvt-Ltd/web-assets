06 — Rule Engine

JSON-based condition trees, operator semantics, reward configuration, and evaluation algorithm

Overview

Rule Engine Architecture

The rule engine evaluates benefit rules against cart and customer context. Rules are stored as structured JSON, cached in Redis, and evaluated in-memory.

Condition Format

Condition Tree Structure

Conditions are stored as JSONB in the benefit_rules.conditions column. They form a tree with AND/OR groups.

JSON Schema

{ "operator": "AND", // top-level group operator "conditions": [ { "field": "cart.total", "op": "gte", "value": 100000 // ₹1000 in paisa }, { "operator": "OR", // nested group "conditions": [ { "field": "customer.tier", "op": "in", "value": ["gold", "platinum"] }, { "field": "customer.tags", "op": "contains", "value": "vip" } ] } ] } // Reads as: cart.total >= ₹1000 AND (customer.tier IN [gold, platinum] OR customer.tags CONTAINS "vip")

Rewards

Reward Configuration

The reward JSONB field in benefit_rules defines what the customer receives. Format varies by benefit type.

Loyalty Coins Reward

{ "type": "earn_coins", "mode": "per_amount", // "per_amount", "per_item", "flat" "coins_per": 1, // 1 coin per ₹X spent "per_amount": 10000, // per ₹100 (paisa) "max_coins": 500, // cap per transaction "multiplier": 2.0 // 2x coins during campaign }

Cashback Reward

{ "type": "cashback", "mode": "percentage", // "percentage", "flat" "percentage": 1000, // 10.00% (basis points for precision) "max_cashback": 50000, // ₹500 cap in paisa "delay": "T+1" // "instant", "T+1", "T+7" }

Discount (Coupon) Reward

{ "type": "discount", "mode": "percentage", // "percentage", "flat" "percentage": 1500, // 15.00% (basis points) "max_discount": 30000, // ₹300 cap in paisa "applies_to": "cart" // "cart", "item", "category" }

Coin Redemption Rate

{ "type": "coin_redemption", "coins_per_rupee": 10, // 10 coins = ₹1 "min_redeem": 100, // minimum 100 coins per redemption "max_redeem_pct": 5000 // max 50% of cart payable with coins (basis points) }
Evaluation

Evaluation Algorithm

func (e *RuleEngine) EvaluateRules( ctx context.Context, evalCtx EvaluationContext, benefitType BenefitType, ) ([]CandidateBenefit, error) { // 1. Load active rules (Redis cached, 5-min TTL) rules, err := e.loadActiveRules(ctx, evalCtx.OrganizationID, evalCtx.WorkspaceID, benefitType) if err != nil { return nil, err } var candidates []CandidateBenefit for _, rule := range rules { // 2. Check temporal validity if !rule.IsActiveAt(evalCtx.Timestamp) { continue } // 3. Check usage limits if rule.MaxUsesTotal > 0 && rule.UsedCount >= rule.MaxUsesTotal { continue } // 4. Evaluate condition tree match, err := e.evaluateConditionTree(rule.Conditions, evalCtx) if err != nil { e.logger.Warn("condition eval error", zap.Error(err)) continue } if !match { continue } // 5. Compute reward amount discount := e.computeReward(rule.Reward, evalCtx) if discount <= 0 { continue } // 6. Build candidate candidates = append(candidates, CandidateBenefit{ ID: rule.ID, Type: rule.BenefitType, Name: rule.Name, DiscountAmount: discount, Priority: rule.Priority, IsStackable: rule.IsStackable, StackGroup: rule.StackGroup, RuleID: rule.ID, }) } return candidates, nil } // evaluateConditionTree recursively evaluates AND/OR condition trees func (e *RuleEngine) evaluateConditionTree( node ConditionNode, ctx EvaluationContext, ) (bool, error) { if node.IsLeaf() { return e.evaluateLeaf(node, ctx) } for _, child := range node.Children { result, err := e.evaluateConditionTree(child, ctx) if err != nil { return false, err } if node.Operator == "AND" && !result { return false, nil // short-circuit AND } if node.Operator == "OR" && result { return true, nil // short-circuit OR } } return node.Operator == "AND", nil }
Caching

Rule Caching Strategy

Redis Cache

Key: rules:{org_id}:{ws_id}:{benefit_type}
TTL: 5 minutes
Invalidation: On rule create/update/delete, cache key is evicted via Redis pub/sub. All service instances subscribe to invalidation events.

Offline Cache

During sync, the POS terminal receives all active rules as JSON. Stored in IndexedDB. Used for local evaluation when offline. Includes a cache_version counter for change detection.
