09 — Offline-First Design

Pre-allocated budgets, cached rules, sync queues, double-spend prevention, and reconciliation

Context

Why Offline Matters

Indian retail POS terminals frequently lose internet connectivity. The elsapiens ecosystem follows an offline-first philosophy where sales must never be blocked by connectivity. elLoyalty extends this principle to benefits.

No Sale Blocked

A customer should be able to use their coins, wallet balance, or coupons even when the terminal is offline. The sale must complete locally.

Approximate, Then Reconcile

Offline operations use cached data and pre-allocated budgets. Exact balances and validation happen on sync. Some operations may be rejected post-facto.

Idempotent Sync

Every offline event has a client-generated UUID (offline_id). If the sync push is retried, the server uses UNIQUE constraints to prevent duplicates.

Budget

Pre-Allocated Offline Budgets

The central innovation: during sync, elLoyalty allocates offline spending budgets per terminal. The terminal can debit up to this budget without server confirmation.

Budget Calculation

Double-Spend

Double-Spend Prevention

Critical Problem: If two terminals both have offline budgets for the same customer (e.g., customer shops at two stores simultaneously), they could both debit coins beyond the actual balance.

Prevention Strategies

1. Budget Per Terminal

Offline budget is allocated per terminal, not per customer globally. If a customer has 500 coins, and two terminals might serve them, each gets 250 coins budget. Configurable split ratio.

2. Pessimistic Budgeting

Default budget is 40% of actual balance. This leaves headroom for multi-terminal scenarios. The percentage is configurable per organization.

3. Idempotent Processing

Every offline transaction has a unique offline_id. The server's UNIQUE (offline_id) constraint prevents duplicate processing if sync is retried.

4. Post-Sync Reconciliation

If total offline debits exceed actual balance, the reconciliation worker creates a "debt" record. The next earning or top-up automatically reduces the debt first.

Server Reconciliation

func (s *SyncService) ProcessPush(ctx context.Context, req SyncPushRequest) (*SyncPushResult, error) { var results []SyncEventResult for _, event := range req.Events { // Idempotency check exists, _ := s.repo.ExistsByOfflineID(ctx, event.OfflineID) if exists { results = append(results, SyncEventResult{ OfflineID: event.OfflineID, Status: "already_processed", }) continue } // Validate: re-check balance, rules, expiry valid, err := s.validateOfflineEvent(ctx, event) if !valid { results = append(results, SyncEventResult{ OfflineID: event.OfflineID, Status: "rejected", Reason: err.Error(), }) continue } // Apply: create real ledger entries from offline event s.applyOfflineEvent(ctx, event) results = append(results, SyncEventResult{ OfflineID: event.OfflineID, Status: "reconciled", }) } return &SyncPushResult{Results: results}, nil }
