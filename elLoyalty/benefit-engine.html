02 — Benefit Engine

Central evaluation pipeline for all five benefit types: Coins, Cashback, Loyalty Cash, Coupons, and Campaigns

Philosophy

Why a Unified Engine?

Without a central engine, each benefit type becomes a silo with its own validation, conflict resolution, and application logic — leading to inconsistent customer experiences and bugs where discounts stack incorrectly.

Single Entry Point

Every benefit evaluation starts here. POS sends a cart → engine returns all applicable benefits, ranked and conflict-resolved. No client-side calculation.

Composable Pipeline

Each benefit type is evaluated by its own provider, but the engine orchestrates the pipeline: evaluate → score → resolve conflicts → present → apply.

Atomic Application

When a customer selects benefits, the engine applies them in a single database transaction. No partial application — either all selected benefits apply or none do.

Providers

Benefit Providers

Each benefit type implements the same BenefitProvider interface, making the engine extensible without modifying core logic.

Provider Interface

// services/benefit_engine.go type BenefitProvider interface { // Type returns the benefit type this provider handles Type() BenefitType // Evaluate returns all possible benefits for the given context Evaluate(ctx context.Context, req EvaluationContext) ([]CandidateBenefit, error) // Apply atomically applies a selected benefit within a transaction Apply(ctx context.Context, tx pgx.Tx, benefit SelectedBenefit) error // Reverse undoes a previously applied benefit (for refunds) Reverse(ctx context.Context, tx pgx.Tx, applicationID string) error } type BenefitType string const ( BenefitCoins BenefitType = "loyalty_coins" BenefitCashback BenefitType = "cashback" BenefitWallet BenefitType = "loyalty_cash" BenefitCoupon BenefitType = "coupon" BenefitCampaign BenefitType = "campaign" )

Five Core Providers

Coins Loyalty Coins Provider

Earn: X coins per ₹Y spent (configurable per category, product, or flat).
Redeem: N coins = ₹M discount. Floor-based: customer chooses how many coins to spend. Minimum redemption threshold enforced.

Cashback Cashback Provider

Earn: Percentage of transaction → credited to wallet after settlement.
Apply: Automatic. Cashback accrues post-sale. Not redeemable mid-checkout but visible in wallet. Delay configurable (instant, T+1, T+7).

Wallet Loyalty Cash Provider

Top-up: Customer adds money to digital wallet (prepaid).
Redeem: Used as a payment method at checkout. Partial or full balance can be applied. Tracks earn vs topup sources separately for accounting.

Coupon Coupon Provider

Validate: Check code → verify expiry, usage limits, min cart, eligible products.
Apply: Flat discount or percentage. Single-use or multi-use. Customer-specific or global. Tracks redemption count per customer.

Campaign Campaign Provider

Meta-provider: Campaign wraps any other benefit type with time-bound, trigger-based, budget-limited constraints. "Buy 3 get 20% off" is a campaign that uses the coupon provider internally.

Engine

BenefitEngine Implementation

type BenefitEngine struct { providers []BenefitProvider ruleEngine *RuleEngine conflictResolver *ConflictResolver walletSvc *WalletService ledgerSvc *LedgerService bus *eventbus.Bus logger *zap.Logger } // Evaluate is the single entry point for all benefit evaluation func (e *BenefitEngine) Evaluate(ctx context.Context, req EvaluationContext) (*EvaluationResult, error) { // 1. Gather customer data wallet, err := e.walletSvc.GetSummary(ctx, req.CustomerID) if err != nil { return nil, fmt.Errorf("gather wallet: %w", err) } // 2. Evaluate each provider var allCandidates []CandidateBenefit for _, p := range e.providers { candidates, err := p.Evaluate(ctx, req) if err != nil { e.logger.Warn("provider evaluation failed", zap.String("type", string(p.Type())), zap.Error(err), ) continue // gracefully degrade; don't block checkout } allCandidates = append(allCandidates, candidates...) } // 3. Score all candidates scored := e.score(allCandidates, req) // 4. Resolve conflicts resolved := e.conflictResolver.Resolve(scored) // 5. Build earn preview earn := e.calculateEarnPreview(req) return &EvaluationResult{ AvailableBenefits: resolved, Recommended: e.pickRecommended(resolved), MaxSavings: e.sumSavings(resolved), EarnPreview: earn, WalletSummary: *wallet, }, nil } // Apply atomically applies selected benefits within a DB transaction func (e *BenefitEngine) Apply(ctx context.Context, req ApplyRequest) (*ApplyResult, error) { // Re-validate: ensure selected benefits are still valid (anti-race) current, err := e.Evaluate(ctx, req.EvaluationContext) if err != nil { return nil, err } validSet := e.validateSelections(req.SelectedBenefitIDs, current.AvailableBenefits) // Begin atomic transaction tx, err := e.pool.Begin(ctx) if err != nil { return nil, err } defer tx.Rollback(ctx) var applied []AppliedBenefit for _, sel := range validSet { provider := e.providerFor(sel.Type) if err := provider.Apply(ctx, tx, sel); err != nil { return nil, fmt.Errorf("apply %s: %w", sel.ID, err) } // Record in immutable ledger e.ledgerSvc.RecordApplication(ctx, tx, sel) applied = append(applied, toApplied(sel)) } if err := tx.Commit(ctx); err != nil { return nil, err } // Publish event for async processing e.bus.Publish(ctx, "benefit.applied", BenefitAppliedEvent{...}) return &ApplyResult{Applied: applied}, nil }
Scoring

Benefit Scoring Algorithm

Each candidate benefit receives a normalized score (0.0 – 1.0) based on multiple factors:

FactorWeightDescription
Discount Value0.40Higher absolute savings = higher score
Rule Priority0.25Admin-assigned priority (1 = highest)
Campaign Boost0.15Active campaigns get a boost
Customer Relevance0.10Tier-specific benefits score higher for matching tier
Recency0.10Newer rules slightly favored to promote fresh offers
func (e *BenefitEngine) score(candidates []CandidateBenefit, ctx EvaluationContext) []CandidateBenefit { maxDiscount := e.findMaxDiscount(candidates) for i := range candidates { c := &candidates[i] s := 0.0 // Discount value (normalized) if maxDiscount > 0 { s += 0.40 * (float64(c.DiscountAmount) / float64(maxDiscount)) } // Priority (inverted: priority 1 → score 1.0) s += 0.25 * (1.0 / float64(max(c.Priority, 1))) // Campaign boost if c.CampaignID != "" { s += 0.15 } // Tier relevance if c.matchesTier(ctx.CustomerTier) { s += 0.10 } c.Score = s } // Sort descending by score sort.Slice(candidates, func(i, j int) bool { return candidates[i].Score > candidates[j].Score }) return candidates }
Graceful

Graceful Degradation

The benefit engine is designed to never block a sale. If evaluation fails, the sale proceeds without benefits.

Provider Failure Isolation

If one provider (e.g., campaign) fails, other providers still return results. The engine logs the error and continues with partial results.

Timeout Protection

Evaluation has a 2-second timeout. If exceeded, the engine returns whatever results it has collected so far, with a partial: true flag.

Offline Fallback

When offline, the POS uses cached rules and pre-allocated budgets to approximate benefits locally. Full reconciliation happens on sync.
