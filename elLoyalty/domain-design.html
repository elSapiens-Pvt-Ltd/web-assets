04 — Domain Design

Bounded contexts, aggregate roots, value objects, and domain events

Bounded Contexts

Six Bounded Contexts

elLoyalty is decomposed into six bounded contexts, each with clear ownership boundaries. Contexts communicate via domain events within the same process.

Context 1

Customer Identity Context

Aggregate Root

LoyaltyCustomer

Owned by elLoyalty. Links to the customer record in elPOS (via customer_id) and person in elAccounts (via person_id). Stores loyalty-specific data: tier, tags, preferences, enrollment date.

Value Object

CustomerTier

Bronze → Silver → Gold → Platinum. Tier is calculated from cumulative spend or points earned over a rolling period. Tier changes trigger notifications.

type LoyaltyCustomer struct { ID string // UUID, primary key OrganizationID string // tenant isolation WorkspaceID string // workspace isolation CustomerID string // ref: elPOS pos_customers.id PersonID string // ref: elAccounts person Tier CustomerTier Tags []string // ["vip", "staff", "wholesale"] TotalSpend int64 // lifetime spend, paisa TotalEarned int // lifetime coins earned TotalRedeemed int // lifetime coins redeemed EnrolledAt time.Time LastActiveAt time.Time Preferences CustomerPreferences } type CustomerTier string const ( TierBronze CustomerTier = "bronze" TierSilver CustomerTier = "silver" TierGold CustomerTier = "gold" TierPlatinum CustomerTier = "platinum" ) type CustomerPreferences struct { SMSNotifications bool EmailNotifications bool WhatsAppOptIn bool AutoRedeemCoins bool // automatically apply coins at checkout }
Context 2

Wallet Context

Aggregate Root

Wallet

Each customer has one wallet with three sub-accounts: CoinBalance, CashbackBalance, and CashBalance (loyalty cash). Balance is always computed from the ledger — never stored directly.

Entity

WalletTransaction

Every wallet mutation is a WalletTransaction: earn, redeem, topup, expire, adjust, reverse. Transactions are immutable — errors are corrected by adding new adjustment entries.

type Wallet struct { ID string CustomerID string OrganizationID string WorkspaceID string // Computed balances (from ledger materialized view) CoinBalance int // current coins CashbackPending int64 // cashback waiting for settlement, paisa CashbackReady int64 // cashback available to spend, paisa CashBalance int64 // prepaid loyalty cash, paisa // Offline budget (pre-allocated for offline use) OfflineCoinBudget int OfflineCashBudget int64 UpdatedAt time.Time } type WalletTransaction struct { ID string WalletID string Type TxType // earn, redeem, topup, expire, adjust, reverse SubAccount SubAccount // coins, cashback, cash Amount int64 // positive = credit, negative = debit ReferenceType string // "sale", "refund", "topup", "expiry" ReferenceID string // sale_id, topup_id, etc. Description string CreatedAt time.Time OfflineID *string // non-nil if created offline }
Context 3

Benefit Engine Context

The Benefit Engine is the orchestrator — it doesn't own persistent data itself, but coordinates across all other contexts. Its aggregate is the BenefitApplication: a record of benefits applied to a specific sale.

type BenefitApplication struct { ID string SaleID string // from elPOS CustomerID string OrganizationID string WorkspaceID string Status AppStatus // pending, applied, reversed, partial_reversed TotalDiscount int64 // paisa Benefits []AppliedBenefit CreatedAt time.Time ReversedAt *time.Time } type AppliedBenefit struct { BenefitID string Type BenefitType RuleID string CampaignID string DiscountAmount int64 Details json.RawMessage // type-specific: coins debited, coupon code, etc. }
Context 4

Rule Engine Context

Aggregate Root

BenefitRule

Defines when and how benefits are granted. Has conditions (cart amount > X, category = Y), constraints (max usage, date range), and rewards (coin multiplier, discount percent). See Rule Engine for full specification.

Entity

RuleCondition

Composable conditions using AND/OR trees. Each condition has a field, operator, and value. Supports cart-level (total, item count), item-level (category, SKU), and customer-level (tier, tags) conditions.

Context 5

Campaign Engine Context

Aggregate Root

Campaign

Time-bound, budget-limited promotions. A campaign wraps one or more BenefitRules with additional constraints: start/end dates, total budget (₹ or quantity), customer enrollment, trigger event.

Entity

CampaignEnrollment

Tracks which customers are enrolled in which campaigns. Some campaigns are auto-enroll (all customers), others require explicit opt-in or invitation.

type Campaign struct { ID string OrganizationID string WorkspaceID string Name string Description string Status CampaignStatus // draft, active, paused, completed, cancelled StartAt time.Time EndAt time.Time BudgetType BudgetType // amount, quantity, unlimited BudgetTotal int64 // max total discount in paisa (or count) BudgetUsed int64 // consumed so far EnrollmentType EnrollmentType // auto, opt_in, invited RuleIDs []string // benefit rules this campaign activates TriggerEvent string // "first_purchase", "birthday", "referral", "" MaxPerCustomer int // max times one customer can benefit CreatedBy string CreatedAt time.Time UpdatedAt time.Time }
Context 6

Transaction Ledger Context

Aggregate Root

LedgerEntry

Every financial event is an immutable ledger entry. The ledger is the source of truth for all balances. See Ledger Design for full specification.

Entity

ExpiryEvent

Scheduled expiry of coins or cashback. A background worker processes expiry events and creates debit ledger entries when coins expire.

Coupons

Coupon: A Composite, Not a Bounded Context

A coupon is not a separate bounded context — it's a composition:

  • Rule Engine owns the coupon's conditions and discount logic (the BenefitRule with type "coupon")
  • Campaign Engine optionally wraps the coupon with time bounds and budget limits
  • Ledger records coupon redemptions
  • Wallet is unaffected (coupons don't touch wallet balances)
type Coupon struct { ID string OrganizationID string WorkspaceID string Code string // unique within org+workspace RuleID string // the benefit rule defining the discount CampaignID *string // optional campaign wrapper MaxUsesTotal int // 0 = unlimited MaxUsesCustomer int // per customer limit UsedCount int AssignedTo []string // empty = global, non-empty = specific customers ValidFrom time.Time ValidUntil time.Time Status CouponStatus // active, exhausted, expired, disabled CreatedAt time.Time }
