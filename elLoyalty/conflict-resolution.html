07 — Conflict Resolution

Priority matrix, stackable vs exclusive benefits, decision trees, and worked scenarios

Problem

Why Conflict Resolution?

When a customer's cart matches multiple benefit rules simultaneously, we must determine which benefits can coexist and which are mutually exclusive. Without conflict resolution, a customer could stack a 20% coupon + 15% cashback + coin redemption and pay nothing.

Stacking Chaos

Without limits, benefits compound: 20% coupon + 10% cashback + loyalty cash = 30%+ off every transaction. Unsustainable for any business.

Customer Confusion

If the POS shows 8 applicable benefits but only 2 can be used together, the UI must clearly communicate why certain combinations are blocked.

Deterministic Outcomes

The same cart + customer + time must always produce the same available benefit set. No randomness, no race conditions between concurrent evaluations.

Priority Matrix

┌────────────────────────────────────────────────────────────────────────┐
│                    BENEFIT PRIORITY MATRIX                              │
│                                                                        │
│  Priority is resolved in this order:                                   │
│                                                                        │
│  1. EXCLUSIVE benefits are evaluated first                             │
│     → Only the highest-scoring exclusive benefit survives              │
│     → All lower-priority exclusive benefits in the same exclusion      │
│       group are removed                                                │
│                                                                        │
│  2. STACKABLE benefits are evaluated next                              │
│     → Stack within the same stack_group                                │
│     → Cross-group stacking allowed unless explicitly excluded           │
│                                                                        │
│  3. PAYMENT-TYPE benefits (coins, wallet) evaluated last               │
│     → These are always additive (they're payment, not discount)        │
│     → Subject to max_redeem_pct cap on cart total                      │
│                                                                        │
│  Within each tier, Priority field (lower number = higher priority)     │
│  breaks ties. If priorities are equal, higher discount amount wins.    │
│                                                                        │
│  ┌──────────────────────────────────────────────────────────────┐      │
│  │ Tier 1: Exclusive Discounts (coupons, exclusive campaigns)   │      │
│  │   → Pick ONE with highest score                              │      │
│  ├──────────────────────────────────────────────────────────────┤      │
│  │ Tier 2: Stackable Discounts (cashback, stackable campaigns)  │      │
│  │   → Keep all that don't conflict with Tier 1 selection       │      │
│  ├──────────────────────────────────────────────────────────────┤      │
│  │ Tier 3: Payment Benefits (coin redeem, wallet pay)           │      │
│  │   → Keep all, subject to cart total cap                      │      │
│  ├──────────────────────────────────────────────────────────────┤      │
│  │ Tier 4: Earning Benefits (coin earn, cashback earn)          │      │
│  │   → Keep all (post-sale, no conflict possible)               │      │
│  └──────────────────────────────────────────────────────────────┘      │
└────────────────────────────────────────────────────────────────────────┘
Algorithm

Resolution Algorithm

func (cr *ConflictResolver) Resolve(candidates []CandidateBenefit) []CandidateBenefit { // Sort by score (descending) sort.Slice(candidates, func(i, j int) bool { return candidates[i].Score > candidates[j].Score }) var result []CandidateBenefit excluded := make(map[string]bool) // benefit IDs excluded by conflict usedGroups := make(map[string]bool) // exclusive groups claimed for _, c := range candidates { // Skip if this benefit was excluded by a higher-priority benefit if excluded[c.ID] { continue } // For exclusive benefits, check if their group is already claimed if !c.IsStackable { group := c.exclusionGroup() if usedGroups[group] { continue // a higher-scoring exclusive in same group already selected } usedGroups[group] = true } // Add to result result = append(result, c) // Mark all benefits this one excludes for _, exID := range c.ExcludesWith { excluded[exID] = true } } return result } // exclusionGroup returns the conflict group for a benefit. // Exclusive benefits in the same group compete; only the best wins. func (c CandidateBenefit) exclusionGroup() string { if c.StackGroup != "" { return c.StackGroup } // Default: each benefit type is its own group return string(c.Type) }
Scenarios

Worked Scenarios

Scenario 1: Coupon + Coins + Cashback

Cart: ₹1,000. Customer has 500 coins (worth ₹50) and code "SAVE15" (15% off, max ₹150).

Matching benefits: Coupon SAVE15 (exclusive, ₹150 off), Coin Redemption (stackable-payment, ₹50 off), 5% Cashback (stackable, ₹50 earned post-sale).

Result: All three are available. Coupon is exclusive within coupon-group (no other coupon), but stacks with coin redemption (payment-type) and cashback (post-sale earning). Customer pays: ₹1,000 - ₹150 (coupon) - ₹50 (coins) = ₹800. Earns ₹40 cashback (5% of ₹800).

Scenario 2: Two Competing Coupons

Cart: ₹2,000. Customer enters "FLAT200" (₹200 off) and also qualifies for auto-coupon "PREMIUM10" (10% off, max ₹300 = ₹200 off).

Result: Both are exclusive in the "coupon" group. PREMIUM10 scores higher (same discount but higher priority = 50 vs 100). Only PREMIUM10 is available. FLAT200 is shown as "unavailable — another coupon already applied" in the UI.

Scenario 3: Campaign + Stackable Cashback

Cart: ₹5,000. Active campaign "Festive Sale" gives 20% off electronics (exclusive). Stackable cashback rule gives 5% cashback on all sales.

Cart items: ₹3,000 electronics + ₹2,000 groceries.

Result: Festive Sale gives ₹600 off (20% of ₹3,000, applies only to electronics items). 5% cashback is stackable and applies to the whole cart post-discount: 5% of ₹4,400 = ₹220 cashback (credited T+1). Customer pays ₹4,400.

Scenario 4: Coin Redemption Exceeds Cap

Cart: ₹500. Customer has 2,000 coins (worth ₹200). Max redemption: 50% of cart.

Result: Max coin discount = 50% of ₹500 = ₹250. Customer can redeem up to ₹200 (their full balance, which is under the cap). If they had ₹300 worth of coins, the cap would limit to ₹250.

Scenario 5: Offline Conflict Resolution

POS offline. Customer wants to use coupon "WEEKEND20" + 100 coins. Cached rules say WEEKEND20 is exclusive and expired yesterday.

Result: Offline cannot validate coupon expiry in real-time (clock may differ). Coupon is tentatively accepted. On sync, elLoyalty re-validates: coupon is expired → rejection event → POS shows "coupon reversed" on next sync. The sale amount is adjusted (or store absorbs the cost per offline policy).
