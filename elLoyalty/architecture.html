01 вҖ” Core Architecture

System overview, technology stack, Clean Architecture layering, and multi-tenancy model

Overview

System Purpose

elLoyalty is a standalone microservice in the elsapiens ecosystem that manages all loyalty and benefit operations: earning, redemption, wallet management, rule evaluation, campaign execution, and audit.

The service is designed as a benefit authority вҖ” every benefit-related decision in the ecosystem flows through elLoyalty. No client application (POS, eCommerce, mobile) makes benefit calculations locally except for cached offline approximations.

Go Module: elsapiens/loyalty

The backend follows the elsapiens convention of Go modules with Clean Architecture, matching the patterns used in elPOS, elbooks, and other ecosystem services.

Layer Responsibilities

Handlers (Transport)

Parse HTTP/gRPC requests, validate input shape, call services, format responses. No business logic. No database access. Handlers are replaceable transport adapters.

Services (Business Logic)

All domain logic lives here: benefit evaluation, rule matching, conflict resolution, wallet operations, ledger entries. Services depend on repository interfaces, not implementations.

Repositories (Data Access)

PostgreSQL queries, Redis caching, transaction management. Repositories implement interfaces defined by services. Unit tests mock repositories via interfaces.

Multi-Tenancy

Tenant Isolation Model

Identical to the elsapiens ecosystem standard. All data rows scoped by (organization_id, workspace_id). JWT claims are the source of truth вҖ” never headers.

Tenant Hierarchy:
  Reseller вҶ’ Workspace вҶ’ Organization вҶ’ Customer

JWT Claims (from elAuth):
  personId      вҶ’ operator identity
  entityId      вҶ’ organization_id
  resellerId    вҶ’ reseller_id (for billing rollup)
  workspaceIds  вҶ’ authorized workspaces

Middleware Flow:
  1. Extract JWT from Cookie or Authorization header
  2. Validate signature (HS256 shared secret)
  3. Extract entityId вҶ’ organization_id (immutable from JWT)
  4. Extract workspaceIds вҶ’ authorized list
  5. Read X-Workspace-ID header вҶ’ must be in workspaceIds
  6. Inject TenantContext into request context

Every SQL query:
  WHERE organization_id = $1 AND workspace_id = $2 AND deleted_at IS NULL

Tenant Context in Go

// internal/middleware/tenant.go type TenantContext struct { OrganizationID string // from JWT entityId вҖ” never from header WorkspaceID string // from X-Workspace-ID, validated against JWT ResellerID string // from JWT resellerId PersonID string // from JWT personId } func RequireAuth(next http.Handler) http.Handler { return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) { claims := validateJWT(r) requestedWS := r.Header.Get("X-Workspace-ID") if !contains(claims.WorkspaceIDs, requestedWS) { respond403(w, "Access denied to workspace") return } ctx := &TenantContext{ OrganizationID: claims.EntityID, WorkspaceID: requestedWS, ResellerID: claims.ResellerID, PersonID: claims.PersonID, } next.ServeHTTP(w, r.WithContext(withTenant(r.Context(), ctx))) }) }
