15 — GraphQL BFF Architecture

Backend-for-Frontend gateway aggregating gRPC services via a single query language

Context

Why a GraphQL Gateway?

elLoyalty exposes gRPC for service-to-service and REST for simple endpoints. A GraphQL BFF sits in front of both, giving each frontend exactly the shape of data it needs — no more, no less.

POS UI Aggregation

A single query fetches customer + wallet + benefits + earn preview. Today this requires 3+ REST calls or custom aggregate endpoints. GraphQL eliminates the waterfall.

Admin Dashboard

Admin pages need deeply nested data — rules with conditions, campaigns with enrollments, customers with transaction history. GraphQL lets the admin UI declare exactly what it needs.

Mobile App APIs

Future mobile wallet app can query a slim subset (balance, recent transactions, tier) without a custom endpoint per screen. Bandwidth-efficient on slow networks.

Not a replacement. GraphQL does NOT replace gRPC (service-to-service) or REST (webhooks, sync endpoints, health checks). It is an additional consumer-facing layer that delegates to internal services.

Resolvers

Resolver Architecture

Resolvers delegate to gRPC clients — they never touch the database. The root resolver holds all service dependencies injected at startup.

Root Resolver

// graph/resolver.go package graph type Resolver struct { LoyaltyClient *grpcclient.LoyaltyClient // elLoyalty gRPC :6010 AccountsClient *grpcclient.AccountsClient // elAccounts gRPC :5000 POSClient *grpcclient.POSClient // elPOS gRPC :6000 Cache *cache.RedisCache }

Query Resolvers

// graph/schema.resolvers.go func (r *queryResolver) Customer(ctx context.Context, id string) (*model.Customer, error) { // 1. Authorize: caller must have terminal or admin role tc := middleware.TenantFromCtx(ctx) if !tc.HasRole("terminal", "store_admin", "org_admin") { return nil, gqlerror.Errorf("forbidden") } // 2. Fetch loyalty customer from elLoyalty gRPC custResp, err := r.LoyaltyClient.GetCustomer(ctx, id) if err != nil { return nil, mapGRPCError(err) } // 3. Enrich with person data from elAccounts (via DataLoader) person, err := dataloader.For(ctx).PersonByID.Load(ctx, custResp.PersonId)() if err != nil { // Graceful degradation: return customer without name r.Logger.Warn("accounts unavailable", "err", err) } return mergeCustomer(custResp, person), nil } func (r *queryResolver) BenefitPreview( ctx context.Context, customerID string, cartItems []*model.CartItemInput, couponCode *string, ) (*model.BenefitPreview, error) { tc := middleware.TenantFromCtx(ctx) if !tc.HasRole("terminal") { return nil, gqlerror.Errorf("forbidden: terminal role required") } // Delegate entirely to elLoyalty gRPC EvaluateCart resp, err := r.LoyaltyClient.EvaluateCart(ctx, &loyaltypb.EvaluateCartRequest{ CustomerId: customerID, Items: toProtoItems(cartItems), CouponCode: derefStr(couponCode), }) if err != nil { return nil, mapGRPCError(err) } return toGQLBenefitPreview(resp), nil }

Mutation Resolvers

func (r *mutationResolver) RedeemBenefits( ctx context.Context, input model.RedeemBenefitsInput, ) (*model.RedemptionResult, error) { tc := middleware.TenantFromCtx(ctx) if !tc.HasRole("terminal") { return nil, gqlerror.Errorf("forbidden") } // Phase 2: Apply benefits via elLoyalty gRPC applyResp, err := r.LoyaltyClient.ApplyBenefits(ctx, &loyaltypb.ApplyBenefitsRequest{ CustomerId: input.CustomerID, SaleId: input.SaleID, BenefitIds: input.SelectedBenefitIds, CartItems: toProtoItems(input.CartItems), OfflineId: derefStr(input.OfflineID), }) if err != nil { return nil, mapGRPCError(err) } // Invalidate DataLoader + Redis cache for this customer's wallet dataloader.For(ctx).WalletByCustomerID.Clear(input.CustomerID) r.Cache.Delete(ctx, fmt.Sprintf("gql:wallet:%s", input.CustomerID)) return toGQLRedemptionResult(applyResp), nil } func (r *mutationResolver) ApplyCoupon( ctx context.Context, input model.ApplyCouponInput, ) (*model.CouponResult, error) { tc := middleware.TenantFromCtx(ctx) if !tc.HasRole("terminal", "store_admin") { return nil, gqlerror.Errorf("forbidden") } resp, err := r.LoyaltyClient.ValidateCoupon(ctx, &loyaltypb.ValidateCouponRequest{ CustomerId: input.CustomerID, CouponCode: input.CouponCode, CartTotalPaisa: int64(input.CartTotalPaisa), }) if err != nil { return &model.CouponResult{Success: false, Error: ptr(err.Error())}, nil } return toGQLCouponResult(resp), nil }

Batching

DataLoader — N+1 Prevention

Without DataLoaders, querying 20 customers would trigger 20 separate wallet gRPC calls. DataLoaders batch and deduplicate within a single request.

WITHOUT DataLoader:                    WITH DataLoader:

  customer(1) → wallet(1)              customer(1) ─┐
  customer(2) → wallet(2)              customer(2) ─┤
  customer(3) → wallet(3)              customer(3) ─┼──▶ GetWallets([1,2,3])
  ...20 gRPC calls                     ...1 batched gRPC call
// graph/dataloader/loaders.go package dataloader import ( "github.com/vikstrous/dataloadgen" ) type Loaders struct { WalletByCustomerID *dataloadgen.Loader[string, *model.Wallet] PersonByID *dataloadgen.Loader[string, *model.Person] CouponsByCustomer *dataloadgen.Loader[string, []*model.Coupon] } func NewLoaders( loyaltyClient *grpcclient.LoyaltyClient, accountsClient *grpcclient.AccountsClient, ) *Loaders { return &Loaders{ WalletByCustomerID: dataloadgen.NewLoader( func(ctx context.Context, keys []string) ([]*model.Wallet, []error) { // Single batched gRPC call for all requested wallets resp, err := loyaltyClient.GetWallets(ctx, keys) if err != nil { errs := make([]error, len(keys)) for i := range errs { errs[i] = err } return nil, errs } return mapWallets(keys, resp), nil }, dataloadgen.WithWait(2*time.Millisecond), // batch window ), PersonByID: dataloadgen.NewLoader( func(ctx context.Context, keys []string) ([]*model.Person, []error) { resp, err := accountsClient.GetPersons(ctx, keys) if err != nil { errs := make([]error, len(keys)) for i := range errs { errs[i] = err } return nil, errs } return mapPersons(keys, resp), nil }, dataloadgen.WithWait(2*time.Millisecond), ), } } // Middleware: inject fresh loaders per request (loaders are NOT reused across requests) func Middleware(loyaltyClient *grpcclient.LoyaltyClient, accountsClient *grpcclient.AccountsClient) func(http.Handler) http.Handler { return func(next http.Handler) http.Handler { return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) { loaders := NewLoaders(loyaltyClient, accountsClient) ctx := context.WithValue(r.Context(), loadersKey, loaders) next.ServeHTTP(w, r.WithContext(ctx)) }) } }
Security

Query Security

Depth Limiting

// internal/middleware/depth.go // Reject queries deeper than maxDepth (default: 7) import "github.com/99designs/gqlgen/graphql/handler/extension" srv := handler.NewDefaultServer(graph.NewExecutableSchema(cfg)) srv.Use(extension.FixedComplexityLimit(1000)) // max complexity cost // Custom depth check in AroundOperations middleware func DepthLimit(maxDepth int) graphql.OperationMiddleware { return func(ctx context.Context, next graphql.OperationHandler) graphql.ResponseHandler { oc := graphql.GetOperationContext(ctx) depth := calculateDepth(oc.Doc.Operations[0].SelectionSet, 0) if depth > maxDepth { return graphql.OneShot(graphql.ErrorResponse(ctx, "query depth %d exceeds maximum %d", depth, maxDepth)) } return next(ctx) } }

Field-Level RBAC

// Directive-based field authorization // schema.graphqls: directive @hasRole(roles: [String!]!) on FIELD_DEFINITION type Customer { id: ID! name: String! wallet: Wallet! transactions(...): TransactionConnection! @hasRole(roles: ["terminal", "store_admin"]) lifetimeSpendPaisa: Int! @hasRole(roles: ["store_admin", "org_admin"]) } // Go directive implementation: func HasRoleDirective(ctx context.Context, obj interface{}, next graphql.Resolver, roles []string) (interface{}, error) { tc := middleware.TenantFromCtx(ctx) for _, r := range roles { if tc.Role == r { return next(ctx) } } return nil, gqlerror.Errorf("access denied: requires role %v", roles) }

Performance

Caching & Persisted Queries

Redis L2 Cache

// cache/redis.go — entity-level caching in the BFF layer type RedisCache struct { client *redis.Client } // Cache resolved entities (not full GraphQL responses) // Short TTL because source of truth is the gRPC services func (c *RedisCache) GetWallet(ctx context.Context, customerID string) (*model.Wallet, error) { key := fmt.Sprintf("gql:wallet:%s", customerID) data, err := c.client.Get(ctx, key).Bytes() if err == redis.Nil { return nil, nil // cache miss } var w model.Wallet json.Unmarshal(data, &w) return &w, nil } func (c *RedisCache) SetWallet(ctx context.Context, customerID string, w *model.Wallet) { key := fmt.Sprintf("gql:wallet:%s", customerID) data, _ := json.Marshal(w) c.client.Set(ctx, key, data, 2*time.Minute) // 2-min TTL }

Persisted Queries for POS

POS terminals send a query hash instead of the full query string. This saves bandwidth on every checkout and prevents arbitrary query injection from the terminal.

// Persisted query flow: // 1. During build, POS app extracts all GraphQL queries → generates hash map // 2. POS sends: POST /graphql { "extensions": { "persistedQuery": { "sha256Hash": "abc123" } } } // 3. BFF resolves hash → full query text from in-memory map // 4. Unknown hashes are rejected (no arbitrary queries from POS) var persistedQueries = map[string]string{ "a1b2c3...": `query CheckoutPreview($cid: ID!, $items: [CartItemInput!]!) { benefitPreview(customerId: $cid, cartItems: $items) { benefits { id type label discountPaisa stackable score } conflicts { benefitId conflictsWith reason } earnPreview { coinsToEarn cashbackPaisa isApproximate } } }`, "d4e5f6...": `mutation Redeem($input: RedeemBenefitsInput!) { redeemBenefits(input: $input) { success totalDiscountPaisa appliedBenefits { type label discountPaisa } updatedWallet { coinBalance cashBalancePaisa } } }`, // ... all POS queries registered at build time } func PersistedQueryMiddleware(next http.Handler) http.Handler { return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) { var body struct { Query string `json:"query"` Extensions struct { PersistedQuery struct { SHA256Hash string `json:"sha256Hash"` } `json:"persistedQuery"` } `json:"extensions"` } // decode, lookup hash, replace body.Query, re-encode // reject if hash not found and client is POS terminal }) }

Server Initialization

// cmd/server/main.go func main() { cfg := config.Load() // gRPC client connections (with circuit breakers) loyaltyConn := grpcclient.Dial(cfg.LoyaltyGRPCAddr) // :6010 accountsConn := grpcclient.Dial(cfg.AccountsGRPCAddr) // :5000 posConn := grpcclient.Dial(cfg.POSGRPCAddr) // :6000 defer loyaltyConn.Close() defer accountsConn.Close() defer posConn.Close() loyaltyClient := grpcclient.NewLoyaltyClient(loyaltyConn) accountsClient := grpcclient.NewAccountsClient(accountsConn) posClient := grpcclient.NewPOSClient(posConn) // Redis cache redisClient := redis.NewClient(&redis.Options{Addr: cfg.RedisAddr}) entityCache := cache.NewRedisCache(redisClient) // GraphQL server resolver := &graph.Resolver{ LoyaltyClient: loyaltyClient, AccountsClient: accountsClient, POSClient: posClient, Cache: entityCache, } srv := handler.NewDefaultServer( graph.NewExecutableSchema(graph.Config{ Resolvers: resolver, Directives: graph.DirectiveRoot{HasRole: HasRoleDirective}, }), ) // Security extensions srv.Use(extension.FixedComplexityLimit(1000)) srv.AroundOperations(DepthLimit(7)) // HTTP router r := chi.NewRouter() r.Use(middleware.Auth(cfg.JWTSecret)) r.Use(middleware.RateLimit(redisClient)) r.Use(dataloader.Middleware(loyaltyClient, accountsClient)) r.Handle("/graphql", middleware.PersistedQuery(srv)) if cfg.PlaygroundEnabled { r.Handle("/", playground.Handler("elLoyalty BFF", "/graphql")) } log.Printf("GraphQL BFF listening on :%s", cfg.Port) http.ListenAndServe(":"+cfg.Port, r) }

Comparison

When to Use GraphQL vs REST vs gRPC

Each protocol serves a distinct purpose in the elLoyalty ecosystem. They coexist — the right tool depends on the consumer and the communication pattern.

GraphQL

  • Consumer: POS UI, Admin Dashboard, Mobile App
  • Pattern: Frontend → BFF
  • Strength: Aggregate multiple services in one round-trip
  • Strength: Client picks exact fields needed
  • Strength: Schema = contract + self-documentation
  • Weakness: Extra hop (BFF → gRPC → service)
  • Weakness: Complex query security (depth, cost)
  • Use when: UI needs data from 2+ services

REST

  • Consumer: Webhooks, Health checks, Sync endpoints, Config API
  • Pattern: External integration
  • Strength: Universal, cacheable (HTTP cache)
  • Strength: Simple for CRUD and file uploads
  • Strength: Easy to debug (curl, browser)
  • Weakness: Over/under-fetching
  • Weakness: Multiple round-trips for related data
  • Use when: Simple single-resource endpoints

gRPC

  • Consumer: elPOS, elbooks, elLoyalty (inter-service)
  • Pattern: Service → Service
  • Strength: Protobuf binary = fast + compact
  • Strength: Streaming, bi-directional
  • Strength: Strict contracts via .proto files
  • Weakness: Not browser-native
  • Weakness: Harder to debug without tooling
  • Use when: Service-to-service, low latency, strict types
