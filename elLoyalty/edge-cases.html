14 — Edge Cases & Failure Modes

Network failures, race conditions, partial sync, clock drift, budget exhaustion, and recovery patterns

Network

Network Failure During Redemption

Edge Case #1: Network drops between Apply and CreditEarnings

POS calls ApplyBenefits (coins debited), network fails, CreditEarnings never reaches server. Customer paid with coins but didn't earn new coins.
Resolution: elPOS queues the CreditEarnings call in the sync queue. On next sync push, the server processes the earn event. The offline_id on the sale ensures idempotency — if the Apply was already processed, it won't duplicate. If Apply itself failed (no server-side record), the sync reconciliation detects the orphan and the POS is notified via the pull response's rejections array.

Edge Case #2: Network drops during ApplyBenefits itself

POS sends ApplyBenefits, server processes and debits wallet, but the response never reaches the POS terminal.
Resolution: POS retries ApplyBenefits with the same offline_id. Server checks: if offline_id already exists in benefit_applications, it returns the original response without re-debiting. This is the core idempotency pattern — every mutating call is safe to retry.
Race

Race Conditions

Edge Case #3: Two terminals apply coins for same customer simultaneously

Terminal A and Terminal B both evaluated 200 coins available. Both send ApplyBenefits to debit 150 coins each (total 300, but only 200 exist).
Resolution: The wallet debit uses PostgreSQL's SELECT ... FOR UPDATE row lock. The first transaction succeeds; the second sees updated balance (50 coins) and either applies partial (50 coins) or rejects the benefit. The handler returns updated available benefits in the response so the POS can re-render.
// Pessimistic lock on wallet for coin debit func (r *WalletRepo) DebitCoins(ctx context.Context, tx pgx.Tx, customerID uuid.UUID, amount int64) error { var balance int64 err := tx.QueryRow(ctx, `SELECT coin_balance FROM wallets WHERE customer_id = $1 FOR UPDATE`, customerID, ).Scan(&balance) if err != nil { return fmt.Errorf("lock wallet: %w", err) } if balance < amount { return ErrInsufficientCoins } _, err = tx.Exec(ctx, `UPDATE wallets SET coin_balance = coin_balance - $1, updated_at = NOW() WHERE customer_id = $2`, amount, customerID, ) return err }

Edge Case #4: Campaign budget exhausts mid-checkout

Campaign has ₹1,000 budget remaining. Terminal A evaluates: "₹500 cashback available." Before Terminal A applies, Terminal B applies ₹800 cashback, exhausting the budget.
Resolution: Campaign budget deduction uses the same FOR UPDATE pattern. Terminal A's apply request sees remaining budget = ₹200. The handler returns a reduced benefit (₹200 instead of ₹500) with a partial: true flag. The POS UI shows "Campaign budget partially available."
Offline

Offline Edge Cases

Edge Case #5: Customer uses expired coupon offline

Coupon WEEKEND20 expires at midnight Saturday. Terminal is offline. Customer uses it at 12:15 AM Sunday (offline time). Cached coupon data says valid.
Resolution: On sync, the server validates the coupon against the sale timestamp. If expired, the sync response includes a rejection. The benefit discount is reversed, creating a debt entry. The POS displays the rejection on next sync. If the amount is small, org can configure auto-forgive thresholds.

Edge Case #6: Clock drift on offline terminal

Terminal's system clock is 30 minutes ahead. Offline sale timestamps are in the future relative to the server. Rules valid "until 6 PM" might be used at 5:45 PM server time but 6:15 PM terminal time.
Resolution: On every sync pull, the server returns server_time. The POS calculates clock offset: offset = server_time - local_time. Offline operations use local_time + offset as the effective timestamp. If offset exceeds 5 minutes, POS shows a warning to the cashier.

Edge Case #7: Terminal offline for days, budget exhausted, no sync

Terminal is offline for 3 days. Offline budget (200 coins) was fully used on day 1. Days 2-3, customers can't use coins at all.
Resolution: This is by design — the offline budget is intentionally conservative. The cashier sees "Coins unavailable offline — budget depleted." For high-traffic stores, the org admin can increase the offline budget. In extreme cases, the POS allows a manual override with store_admin PIN, creating a "trust debit" that's validated on sync.
Refunds

Refund & Reversal Edge Cases

Edge Case #8: Partial refund with mixed benefits

Sale: ₹2,000. Applied: 100 coins (₹100 off) + cashback (₹60 earned) + coupon FLAT200 (₹200 off). Customer returns one item worth ₹800. How to reverse?
Resolution: Proportional reversal. Refund ratio = ₹800 / ₹2,000 = 40%. Coins reclaimed: 40 coins (40% of 100). Cashback clawed back: ₹24 (40% of ₹60). Coupon: no reversal (single-use coupons are not proportionally refundable — the coupon stays consumed). The ledger creates reversal entries for each reversed amount.

Edge Case #9: Refund after coins already spent

Customer earned 200 coins from Sale A. Customer spends 200 coins on Sale B. Sale A is fully refunded. Coins should be reclaimed but balance is now 0.
Resolution: The refund creates a "debt" ledger entry: coin_balance goes to -200. Next time the customer earns coins, they offset the debt first. The customer's wallet shows: "0 available (200 coins pending recovery)." The system never blocks future transactions — it just reduces future earnings until the debt is cleared.
Data

Data Integrity Edge Cases

Edge Case #10: Duplicate sync push (network timeout + retry)

Terminal pushes 5 offline events. Server processes all 5, but the HTTP response times out. Terminal retries the same 5 events.
Resolution: Each event has a unique offline_id. The server checks ExistsByOfflineID before processing. All 5 are returned as "already_processed". No double-crediting, no double-debiting. The UNIQUE constraint on offline_id (with NULLS NOT DISTINCT) is the database-level safety net.

Edge Case #11: Rule change while offline terminals have cached old rules

Admin changes "2x coins on electronics" to "1x coins" at 3 PM. Terminal was offline since 1 PM. Customer gets 2x coins offline at 4 PM.
Resolution: On sync reconciliation, the server re-evaluates the earn using current rules and the sale timestamp. If the offline earn (200 coins) differs from the correct earn (100 coins), a correction entry is created: -100 coins. The customer is notified via elmessagehub. This "eventual consistency" is the trade-off for offline capability.
Summary

Design Principles for Edge Cases

Eventual Consistency

Offline operations are approximate. The server is the source of truth. Every offline action is reconciled and potentially corrected on sync.

Idempotency Everywhere

Every mutating operation accepts an offline_id. Database UNIQUE constraints + application-level checks guarantee at-most-once processing.

Debt Over Rejection

When a reversal can't fully reclaim coins/cash (already spent), create a debt instead of blocking the customer. Future earnings offset the debt automatically.

Proportional Reversal

Partial refunds reverse benefits proportionally. Single-use items (coupons) follow different rules than accrual items (coins, cashback).

Pessimistic Locking for Money

All wallet mutations use SELECT ... FOR UPDATE. Optimistic concurrency is not safe when money is involved — PostgreSQL row locks are the foundation.

Graceful Degradation

If a downstream service fails, loyalty features degrade rather than block checkout. Circuit breakers prevent cascade failures.
